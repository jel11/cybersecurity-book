## 7.4.1 Зачем нужны отчёты об инцидентах

### Цели документирования

```
ДЛЯ ЧЕГО НУЖЕН ОТЧЁТ ОБ ИНЦИДЕНТЕ:

1. ЮРИДИЧЕСКАЯ ЗАЩИТА
   └── Доказательство должной осторожности (due diligence)
   └── Основа для судебного преследования атакующих
   └── Требование регуляторов (GDPR, PCI DSS, HIPAA)

2. БИЗНЕС-ЦЕННОСТЬ
   └── Обоснование инвестиций в безопасность
   └── Демонстрация эффективности SOC
   └── Основа для страховых выплат (Cyber Insurance)

3. ТЕХНИЧЕСКИЕ ЦЕЛИ
   └── Документирование timeline и IoC для команды
   └── Основа для улучшения детекций и плейбуков
   └── База знаний для будущих инцидентов

4. КОММУНИКАЦИЯ
   └── Информирование руководства об угрозах
   └── Уведомление регуляторов (обязательно по закону)
   └── Уведомление пострадавших клиентов
```

### Типы отчётов

```
ТИПЫ ОТЧЁТОВ ОБ ИНЦИДЕНТАХ:

┌────────────────────┬──────────────────────────────────────────┐
│ Оперативный        │ Создаётся в процессе инцидента.          │
│ (Incident Ticket)  │ Живой документ. Тикет в TheHive/JIRA.   │
│                    │ Аудитория: команда реагирования          │
├────────────────────┼──────────────────────────────────────────┤
│ Технический        │ Детальный технический анализ после       │
│ (Technical Report) │ закрытия. Временная линия, IOC,         │
│                    │ техники MITRE ATT&CK.                    │
│                    │ Аудитория: ИБ команда, ИТ               │
├────────────────────┼──────────────────────────────────────────┤
│ Исполнительное     │ Краткое резюме для не-технической        │
│ резюме (Executive  │ аудитории. Без технических деталей.     │
│ Summary)           │ Аудитория: CISO, CEO, Board             │
├────────────────────┼──────────────────────────────────────────┤
│ Post-Mortem /      │ Анализ причин и уроков.                  │
│ Lessons Learned    │ Аудитория: вся команда                  │
├────────────────────┼──────────────────────────────────────────┤
│ Регуляторный       │ Уведомление регулятора (GDPR, ЦБ РФ).  │
│ (Regulatory)       │ Стандартная форма. Юридически значим.   │
└────────────────────┴──────────────────────────────────────────┘
```

## 7.4.2 Технический отчёт об инциденте

### Полная структура технического отчёта

```
СТРУКТУРА ТЕХНИЧЕСКОГО ОТЧЁТА:

1. ТИТУЛЬНАЯ СТРАНИЦА
   ├── Название: "Отчёт об инциденте: Ransomware — INCIDENT-2024-0115"
   ├── Классификация: CONFIDENTIAL / TLP:AMBER
   ├── Дата создания и дата последнего обновления
   ├── Автор(ы) отчёта
   └── Статус: Draft / Final

2. КРАТКОЕ РЕЗЮМЕ (1 страница)
   ├── Тип инцидента
   ├── Дата и период
   ├── Затронутые системы (кол-во)
   ├── Влияние на бизнес
   └── Статус (закрыт/в работе)

3. ТЕХНИЧЕСКИЕ ДЕТАЛИ
   ├── Вектор атаки (Initial Access)
   ├── Временная линия
   ├── Использованные техники (MITRE ATT&CK)
   ├── Затронутые системы (детально)
   └── Масштаб компрометации

4. ИНДИКАТОРЫ КОМПРОМЕТАЦИИ (IOC)
   ├── IP адреса
   ├── Домены
   ├── Хэши файлов
   ├── Registry ключи
   └── Сигнатуры YARA/Sigma

5. ФОРЕНЗИЧЕСКИЕ ДАННЫЕ
   ├── Источники данных
   ├── Методология анализа
   └── Ключевые находки

6. ПРЕДПРИНЯТЫЕ ДЕЙСТВИЯ
   └── Хронология реагирования

7. РЕКОМЕНДАЦИИ
   ├── Краткосрочные (немедленно)
   ├── Среднесрочные (1-3 месяца)
   └── Долгосрочные (стратегические)

8. ПРИЛОЖЕНИЯ
   ├── Полные логи
   ├── PCAP файлы
   └── Скриншоты
```

```markdown

## 1. Краткое резюме

признаки активности ransomware на рабочей станции WS-JOHN01 пользователя

ограничить ущерб двумя рабочими станциями и одним файловым сервером.

копии). Простой бизнеса: 6 часов.

**Корневая причина:** Пользователь открыл вредоносное вложение из фишингового письма
и запустил замаскированный исполняемый файл.

## 3. Технический анализ

### 3.1 Вектор атаки (Initial Access)

Атака начиналась с целевого фишингового письма:

```
From: hr@company-update.com (подделка под HR отдел)
Reply-To: attacker@protonmail.com
Subject: Обновление политики HR — требуется подпись
Attachment: HR_Policy_2024.docx.exe (маскировка под Word документ)
```

Заголовки письма:
- SPF: FAIL (домен company-update.com не авторизован)
- DKIM: Не подписано
- Return-Path: @smtp.attacker-smtp.ru

### 3.2 Техники по MITRE ATT&CK

| ID | Техника | Описание |
|----|---------|----------|
| T1566.001 | Spear Phishing (Attachment) | Фишинговое письмо с вложением |
| T1204.002 | User Execution: Malicious File | Пользователь запустил файл |
| T1059.001 | PowerShell | Загрузка payload через PowerShell |
| T1562.001 | Disable Security Tools | Отключение Windows Defender |
| T1055 | Process Injection | Инъекция в svchost.exe |
| T1078 | Valid Accounts | Использование LSASS дампа |
| T1021.002 | SMB/Windows Admin Shares | Lateral movement через SMB |
| T1486 | Data Encrypted for Impact | Шифрование файлов (ransomware) |
| T1490 | Inhibit System Recovery | Удаление теневых копий VSS |

### 3.3 Технические детали malware

**Файл:** HR_Policy_2024.docx.exe
- **SHA256:** `a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3`
- **MD5:** `e10adc3949ba59abbe56e057f20f883e`
- **Размер:** 1,245,184 байт
- **Тип:** PE32+ executable, upx-packed
- **VirusTotal:** 67/94 (LockBit.3.0, Ransom.Win64.LockBit)

**Поведение:**
1. Распаковывается в `%TEMP%\svchost32.exe`
2. Создаёт Scheduled Task: `Microsoft\Windows\Update\UpdateCheck`
3. Дампит LSASS память в `%TEMP%\debug.dmp`
4. Подключается к C2: 185.220.101.47:443 (TLS)
5. Удаляет теневые копии: `vssadmin delete shadows /all /quiet`
6. Шифрует файлы, добавляет расширение `.locked`

## 7.4.3 Исполнительное резюме (Executive Summary)

Исполнительное резюме — это отдельный документ (1-2 страницы максимум) для CISO, CEO, Board Directors. Они не технари — им нужен деловой язык.

```
ПРАВИЛА EXECUTIVE SUMMARY:

✓ Максимум 1-2 страницы
✓ Никаких технических деталей (CVE, хэши, MITRE)
✓ Деловой язык (влияние на бизнес, деньги, риски)
✓ Конкретные числа: сколько систем, сколько часов, сколько стоит
✓ Чёткие рекомендации с приоритетами
✓ Акцент на том, что было сделано правильно
✗ Никаких оправданий и поиска виноватых
✗ Никаких технических деталей атаки
✗ Никаких "мы не знаем/непонятно"
```

```markdown
---
**КОНФИДЕНЦИАЛЬНО**
---

## Ransomware-атака | 15–18 января 2024

---

атаки, что привело к заражению 2 рабочих станций и 1 файлового сервера

## 7.4.4 Post-Mortem (Lessons Learned)

Post-mortem — это встреча и документ, где команда честно обсуждает, что пошло не так и как предотвратить повторение.

### Принципы blameless post-mortem

```
BLAMELESS (БЕЗВИННЫЙ) POST-MORTEM:

Мы исходим из того, что:
• Каждый действовал из лучших побуждений
• Каждый сделал всё возможное с тем, что знал в тот момент
• Ошибки — результат сбоев в системе, не людей
• Цель — улучшить систему, не наказать людей

ВОПРОСЫ ДЛЯ ОБСУЖДЕНИЯ:
• Что мы сделали хорошо? (обязательно начать с этого!)
• Что можно было сделать лучше?
• Что нас удивило?
• Что нужно изменить в процессах/инструментах?
```

               Alexey Sidorov (SOC Analyst), IT Director

## Что сработало хорошо

✅ **Быстрое обнаружение:** SIEM алерт сработал через 2 часа после
   запуска malware — это в пределах нашего SLA для P1 инцидентов.

   30 минут. Никакой путаницы в командной цепочке.

✅ **Резервные копии:** Ежедневные резервные копии позволили восстановить
   1.2TB данных без уплаты выкупа.

✅ **Документирование:** Временная линия велась в реальном времени,
   что упростило расследование.

## Что пошло не так / Что можно улучшить

### 1. Алерт был проигнорирован на 2 часа
**Что произошло:** SIEM сгенерировал алерт о подозрительном PowerShell
в 09:31, но дежурный аналитик закрыл его как False Positive.

**Почему:** Алерт не имел дополнительного контекста, а аналитик
был перегружен (одновременно работал над P2 инцидентом).

**Action Item:**
- Обновить правило SIEM: добавить корреляцию PowerShell + outbound connection = P1
- Внедрить автоматическое обогащение алертов (IP → AbuseIPDB)
- Пересмотреть нагрузку на дежурного аналитика

**Ответственный:** Maria Ivanova  **Срок:** 2024-02-01

### 2. Нет изоляции на уровне сети (только EDR)
**Что произошло:** Изоляция через EDR заняла 8 минут. За это время
malware распространился на FILESERVER01.

**Почему:** Нет автоматической сетевой изоляции при обнаружении ransomware.

**Action Item:**
- Настроить автоматическое помещение в quarantine VLAN при EDR-детекции ransomware
- Провести тест автоматической изоляции в следующем квартале

**Ответственный:** Alexey Sidorov  **Срок:** 2024-03-01

### 3. Пользователь прошёл через фишинг
**Что произошло:** john.doe открыл подозрительное письмо
несмотря на политику безопасности.

**Почему:** Последний тренинг был год назад. Письмо было убедительным
(от имени HR).

**Action Item:**
- Провести фишинг-тренинг для всех сотрудников (1 месяц)
- Ввести ежеквартальные симулированные фишинговые тесты
- Упростить процесс репортинга подозрительных писем (кнопка в Outlook)

**Ответственный:** HR + SOC Team  **Срок:** 2024-02-15

### 4. Нет offline резервных копий
**Что произошло:** Резервные копии хранились на сетевом ресурсе,
который также мог быть зашифрован (к счастью — не был).

**Почему:** Нет политики immutable/offline backup.

**Action Item:**
- Внедрить 3-2-1 правило резервного копирования (3 копии, 2 носителя, 1 офлайн)
- Рассмотреть cloud backup (immutable storage)

**Ответственный:** IT Director  **Срок:** 2024-03-15

## 7.4.7 Автоматизация создания отчётов

```python
#!/usr/bin/env python3
"""
incident_report_generator.py
Генерирует markdown отчёт об инциденте из данных тикета

pip install jinja2 python-dateutil
"""

from jinja2 import Template
from datetime import datetime
import json

REPORT_TEMPLATE = """
# Технический отчёт: {{ incident_id }}

| Параметр | Значение |
|----------|----------|
| ID | {{ incident_id }} |
| Тип | {{ incident_type }} |
| Критичность | {{ severity }} |
| Дата начала | {{ start_time }} |
| Дата закрытия | {{ end_time | default('В работе') }} |
| Аналитик | {{ analyst }} |

## Краткое резюме
{{ summary }}

## Временная линия

| Время (UTC) | Событие | Источник |
|-------------|---------|----------|
{% for event in timeline -%}
| {{ event.time }} | {{ event.description }} | {{ event.source }} |
{% endfor %}

## Индикаторы компрометации (IOC)

{% if iocs.ips %}
### IP адреса
{% for ip in iocs.ips %}
- `{{ ip }}`
{% endfor %}
{% endif %}

{% if iocs.domains %}
### Домены
{% for domain in iocs.domains %}
- `{{ domain }}`
{% endfor %}
{% endif %}

{% if iocs.hashes %}
### Хэши файлов
{% for hash in iocs.hashes %}
- `{{ hash }}`
{% endfor %}
{% endif %}

## Рекомендации

{% for rec in recommendations %}
{{ loop.index }}. **[{{ rec.priority }}]** {{ rec.action }} — Срок: {{ rec.deadline }}
{% endfor %}

---
*Отчёт сгенерирован: {{ generated_at }}*
*Автор: {{ analyst }}*
"""

def generate_incident_report(incident_data: dict) -> str:
    """Генерация markdown отчёта из данных инцидента"""
    template = Template(REPORT_TEMPLATE)
    incident_data["generated_at"] = datetime.utcnow().strftime("%Y-%m-%d %H:%M UTC")
    return template.render(**incident_data)

# Пример данных инцидента
incident_data = {
    "incident_id": "INCIDENT-2024-0115-001",
    "incident_type": "Ransomware (LockBit 3.0)",
    "severity": "P1 - Critical",
    "start_time": "2024-01-15 09:23 UTC",
    "end_time": "2024-01-18 14:00 UTC",
    "analyst": "Maria Ivanova",
    "summary": "Ransomware-атака через фишинговое письмо. Изолированы 2 хоста и 1 сервер. Данные восстановлены из резервной копии.",
    "timeline": [
        {"time": "09:23", "description": "Фишинговое письмо получено", "source": "Email gateway"},
        {"time": "09:31", "description": "Пользователь открыл вложение", "source": "Windows Event 4688"},
        {"time": "11:17", "description": "СИЕМ алерт: ransomware", "source": "ELK SIEM"},
        {"time": "11:25", "description": "Изоляция хостов", "source": "SOC Action"},
    ],
    "iocs": {
        "ips": ["185.220.101.47", "91.108.56.23"],
        "domains": ["company-update.com", "cdn-update.net"],
        "hashes": [
            "SHA256: a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3"
        ]
    },
    "recommendations": [
        {"priority": "HIGH", "action": "Внедрить MFA для всех сотрудников", "deadline": "2024-02-01"},
        {"priority": "HIGH", "action": "Провести фишинг-тренинг", "deadline": "2024-02-15"},
        {"priority": "MEDIUM", "action": "Offline резервные копии", "deadline": "2024-03-15"},
    ]
}

report = generate_incident_report(incident_data)
print(report)

# Сохраняем в файл
with open(f"report_{incident_data['incident_id']}.md", "w", encoding="utf-8") as f:
    f.write(report)
print(f"[+] Report saved to report_{incident_data['incident_id']}.md")
```

## 📌 Итоги главы

- Отчёты об инцидентах — не бюрократия, а ценный инструмент для защиты, улучшения и коммуникации
- **Технический отчёт**: детальная временная линия, MITRE ATT&CK маппинг, IOC, конкретные рекомендации
- **Executive Summary**: деловой язык, влияние на бизнес, максимум 2 страницы — для CISO/CEO
- **Post-Mortem**: blameless-подход, конкретные action items с ответственными и сроками
- **Регуляторные уведомления**: GDPR — 72 часа; 187-ФЗ (КИИ) — немедленно
- Метрики MTTD/MTTR/MTTC — объективная оценка эффективности SOC
- Хороший отчёт: конкретные числа, факты, без оправданий, с чёткими следующими шагами
