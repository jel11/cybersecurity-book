# Глава 7.1: Жизненный цикл инцидента по NIST

## 🎯 Цели главы

- Понять фреймворк NIST SP 800-61 для реагирования на инциденты
- Освоить все фазы жизненного цикла инцидента
- Научиться правильно классифицировать и приоритизировать инциденты
- Разобраться в организационной структуре IR-команды
- Изучить метрики эффективности реагирования

## 7.1.1 Что такое инцидент безопасности

### Определение

**Инцидент информационной безопасности** — это нарушение или угроза нарушения политики безопасности, политики допустимого использования или стандартных практик безопасности.

```
Событие (Event) vs Инцидент (Incident):

СОБЫТИЕ — любое наблюдаемое действие в системе:
  • Пользователь вошёл в систему в 3:00 ночи
  • Файрвол заблокировал пакет
  • Антивирус сделал предупреждение

ИНЦИДЕНТ — событие с негативными последствиями:
  • Пользователь вошёл в систему в 3:00 ночи и скачал 50GB данных
  • Файрвол заблокировал атаку сканирования на 1000 портов
  • Антивирус обнаружил и не смог удалить ransomware

Не каждое событие — инцидент.
Каждый инцидент — событие.
```

### Типы инцидентов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Malware** | Заражение вредоносным ПО | Ransomware, trojan, worm |
| **Unauthorized Access** | Несанкционированный доступ | Взлом учётной записи, SQL injection |
| **DDoS** | Отказ в обслуживании | Атака на веб-сервер |
| **Data Breach** | Утечка данных | Кража клиентской базы |
| **Insider Threat** | Внутренняя угроза | Сотрудник копирует данные |
| **Phishing** | Фишинг | Email с вредоносным вложением |
| **Privilege Escalation** | Повышение привилегий | Получение root-доступа |
| **Supply Chain** | Атака цепочки поставок | Скомпрометированное ПО |

## 7.1.2 Фреймворк NIST SP 800-61

NIST (National Institute of Standards and Technology) разработал стандарт **SP 800-61 "Computer Security Incident Handling Guide"** — наиболее широко используемый фреймворк реагирования на инциденты.

### Четыре фазы жизненного цикла

```
┌─────────────────────────────────────────────────────────────────────┐
│              ЖИЗНЕННЫЙ ЦИКЛ ИНЦИДЕНТА (NIST SP 800-61)              │
├─────────────┬──────────────┬──────────────┬─────────────────────────┤
│             │              │              │                         │
│ Подготовка  │ Обнаружение  │ Сдерживание  │ Восстановление          │
│ (Prepare)   │ и анализ     │ ликвидация   │ и извлечение            │
│             │ (Detect &    │ (Containment,│ уроков                  │
│             │  Analyze)    │  Eradication │ (Recovery &             │
│             │              │  & Recovery) │  Post-Incident)         │
│             │              │              │                         │
└─────────────┴──────────────┴──────────────┴─────────────────────────┘
         ↑                                              │
         └──────────────────────────────────────────────┘
                    (Непрерывное улучшение)
```

## 7.1.3 Фаза 1: Подготовка (Preparation)

Подготовка — самая важная фаза. Реагировать на инцидент без подготовки — всё равно что тушить пожар без пожарного оборудования.

### Что входит в подготовку

```
ОРГАНИЗАЦИОННАЯ ПОДГОТОВКА:
├── Создание IR-команды (CIRT/CSIRT)
├── Определение ролей и ответственности
├── Разработка политик реагирования
├── Написание плейбуков (Playbooks)
├── Определение цепочки эскалации
└── Контакты: юристы, PR, руководство, регуляторы

ТЕХНИЧЕСКАЯ ПОДГОТОВКА:
├── SIEM (Splunk, ELK, QRadar)
├── EDR (CrowdStrike, Defender for Endpoint)
├── Ticket-система для инцидентов (TheHive, JIRA)
├── Инфраструктура форензики (Volatility, FTK)
├── Изолированная sandbox-среда
├── Jump-серверы для расследования
└── Резервный канал связи (Signal, не корпоративный)

ПРОЦЕССНАЯ ПОДГОТОВКА:
├── Плейбуки для каждого типа инцидента
├── Матрица приоритизации (Impact × Urgency)
├── SLA для каждого уровня критичности
├── Шаблоны отчётов
└── Договоры с внешними IR-компаниями
```

### Матрица классификации по критичности

```
МАТРИЦА ПРИОРИТИЗАЦИИ ИНЦИДЕНТОВ:

              │ Impact: LOW │ Impact: MED │ Impact: HIGH │
──────────────┼─────────────┼─────────────┼──────────────┤
Urgency: LOW  │   P4 (Low)  │  P3 (Med)   │  P2 (High)   │
──────────────┼─────────────┼─────────────┼──────────────┤
Urgency: MED  │   P3 (Med)  │  P2 (High)  │  P1 (Crit)   │
──────────────┼─────────────┼─────────────┼──────────────┤
Urgency: HIGH │   P2 (High) │  P1 (Crit)  │  P1 (Crit)   │
──────────────┴─────────────┴─────────────┴──────────────┘

P1 Critical: SLA = 15 минут на ответ, круглосуточно
P2 High:     SLA = 1 час на ответ, рабочие часы + on-call
P3 Medium:   SLA = 4 часа на ответ, рабочие часы
P4 Low:      SLA = 24 часа на ответ, рабочие часы
```

## 7.1.4 Фаза 2: Обнаружение и анализ (Detection & Analysis)

### Источники обнаружения инцидентов

```
КАК ОБНАРУЖИВАЮТСЯ ИНЦИДЕНТЫ:

Автоматические системы (60-70%):
├── SIEM алерты (корреляция событий)
├── EDR/AV детекции
├── IDS/IPS алерты
├── Файрвол блокировки
└── DLP срабатывания

Люди (20-30%):
├── Пользователи сообщают о странном поведении
├── SOC-аналитик замечает аномалию
├── Системный администратор обнаруживает изменения
└── Threat hunting находит скрытую активность

Внешние источники (5-10%):
├── Уведомление от CERT/CISA
├── Информация от партнёров
├── Репорт от исследователя безопасности
└── Статья в новостях (worst case!)
```

### Процесс анализа инцидента

```python
# Алгоритм анализа инцидента (псевдокод)

def analyze_incident(alert):
    """
    Первичный анализ инцидента
    """

    # Шаг 1: Сбор начальных данных
    context = {
        "timestamp": alert.time,
        "source_ip": alert.src_ip,
        "destination": alert.dst_ip,
        "affected_systems": [],
        "alert_type": alert.rule_name,
        "raw_logs": []
    }

    # Шаг 2: Обогащение данных
    context["ip_reputation"] = check_virustotal(alert.src_ip)
    context["geo_location"] = geoip_lookup(alert.src_ip)
    context["domain_info"] = whois_lookup(alert.src_domain)
    context["threat_intel"] = check_misp(alert.iocs)

    # Шаг 3: Определение False Positive vs True Positive
    if is_false_positive(alert, context):
        close_alert("False Positive", reason=context)
        return

    # Шаг 4: Классификация инцидента
    incident_type = classify_incident(alert)
    severity = calculate_severity(incident_type, context)

    # Шаг 5: Создание тикета
    ticket = create_incident_ticket(
        type=incident_type,
        severity=severity,
        context=context,
        playbook=get_playbook(incident_type)
    )

    # Шаг 6: Эскалация по приоритету
    escalate_if_needed(ticket, severity)

    return ticket
```

### 5 вопросов первичного анализа

```
ЧТО ПРОИЗОШЛО? (What)
  → Тип инцидента: malware, unauthorized access, DDoS?
  → Какие системы затронуты?

КТО? (Who)
  → Внутренний или внешний актор?
  → Атрибуция (если возможно): APT группа, скрипт-кидди?
  → Затронуты ли определённые пользователи?

КОГДА? (When)
  → Время начала инцидента (дата первого следа)
  → Как долго атака происходила до обнаружения?
  → Временная линия событий

ГДЕ? (Where)
  → Scope: 1 хост, подсеть, весь домен?
  → Охвачены ли продакшен-системы?
  → Есть ли признаки lateral movement?

КАК? (How)
  → Вектор атаки (initial access): фишинг, эксплойт, VPN?
  → Техники по MITRE ATT&CK
  → Какие уязвимости использованы?
```

## 7.1.5 Фаза 3: Сдерживание, ликвидация и восстановление

### 3.1 Сдерживание (Containment)

Цель: остановить распространение, не уничтожая доказательства.

```
КРАТКОСРОЧНОЕ СДЕРЖИВАНИЕ (немедленно):
├── Изоляция заражённых хостов от сети
│   ├── EDR: изоляция хоста одной кнопкой
│   ├── Сеть: VLAN quarantine или отключение порта
│   └── НИКОГДА не выключать компьютер (теряем RAM)
├── Блокировка скомпрометированных учётных записей
├── Блокировка C2-серверов на файрволе/DNS
└── Уведомление пользователей о запрете использования

ДОЛГОСРОЧНОЕ СДЕРЖИВАНИЕ (1-24 часа):
├── Создание forensic-образа дисков (dd, FTK Imager)
├── Дамп оперативной памяти (Volatility)
├── Сбор всех логов
├── Развёртывание временных систем (для бизнеса)
└── Усиление мониторинга на периметре
```

```bash
# Пример: Изоляция хоста через EDR API (CrowdStrike)
# (концептуально - реальный API требует ключей)

curl -X POST "https://api.crowdstrike.com/devices/entities/devices-actions/v2" \

    "action_name": "contain",

# Пример: Блокировка пользователя в AD (PowerShell)

Get-ADUser "john.doe" | Set-ADUser -LockedOut $true

# Дамп памяти для форензики
# На Windows:
# WinPmem или DumpIt (запустить от админа)

# На Linux:
dd if=/proc/kcore of=/mnt/usb/memory.raw bs=1M
# или LiME модуль для точного дампа
```

### 3.2 Ликвидация (Eradication)

```
ЧТО НУЖНО УДАЛИТЬ/ИСПРАВИТЬ:

Malware:
├── Полное удаление malware и его компонентов
├── Удаление persistence mechanisms:
│   ├── Registry Run keys
│   ├── Scheduled tasks
│   ├── Services
│   ├── Startup folders
│   └── WMI subscriptions
├── Откат изменений конфигурации
└── Удаление backdoor-аккаунтов

Уязвимости:
├── Патчинг эксплуатированных уязвимостей
├── Смена паролей всех скомпрометированных аккаунтов
├── Ротация API ключей и сертификатов
├── Исправление мисконфигураций
└── Закрытие лишних портов/сервисов

ВАЖНО: Перед ликвидацией сохранить форензические данные!
```

### 3.3 Восстановление (Recovery)

```
ПОРЯДОК ВОССТАНОВЛЕНИЯ:

1. ПРОВЕРКА ЧИСТОТЫ
   └── Подтверждение полного удаления malware
   └── Повторное сканирование EDR/AV

2. ВОССТАНОВЛЕНИЕ ИЗ БЕКАПА
   └── Использовать ЧИСТЫЙ бекап (до заражения!)
   └── Проверить целостность бекапа
   └── Поэтапное восстановление

3. HARDENING
   └── Применить дополнительные меры защиты
   └── Включить дополнительное логирование
   └── Обновить правила SIEM/EDR

4. МОНИТОРИНГ
   └── Усиленный мониторинг в течение 30+ дней
   └── Проверка на признаки повторного заражения
   └── Threat hunting по IOC

5. ВОЗВРАТ В ПРОДАКШЕН
   └── Поэтапный возврат (не всё сразу)
   └── Тестирование функциональности
   └── Уведомление пользователей
```

## 7.1.6 Фаза 4: Действия после инцидента (Post-Incident Activity)

### После-инцидентный анализ (Post-Mortem / Lessons Learned)

Проводится в течение 2 недель после инцидента. Это не поиск виноватых — это поиск улучшений.

```
СТРУКТУРА POST-MORTEM:

1. ХРОНОЛОГИЯ
   └── Детальная временная линия: от initial access до закрытия

2. ЧТО СРАБОТАЛО ХОРОШО
   └── EDR быстро обнаружил malware
   └── Плейбук помог чётко действовать
   └── Коммуникация была эффективной

3. ЧТО ПОШЛО НЕ ТАК
   └── Алерт был, но проигнорирован (2 часа)
   └── Нет изоляции на уровне сети
   └── Нет актуального бекапа серверов

4. ROOT CAUSE (Корневая причина)
   └── Почему стало возможным?
   └── Уязвимость, мисконфигурация, человеческий фактор?

5. ACTION ITEMS (Что делаем)
   └── [Ответственный] Патч MS17-010 до 01.02.2024
   └── [Ответственный] Настроить сетевую изоляцию в EDR
   └── [Ответственный] Обновить плейбук по ransomware
   └── [Ответственный] Провести phishing-тренинг для HR

6. МЕТРИКИ ИНЦИДЕНТА
   └── MTTD: Mean Time to Detect (время до обнаружения)
   └── MTTR: Mean Time to Respond (время до реагирования)
   └── MTTC: Mean Time to Contain (время до изоляции)
   └── MTTRE: Mean Time to Remediate (время до устранения)
```

### Метрики SOC/IR

```
КЛЮЧЕВЫЕ МЕТРИКИ ЭФФЕКТИВНОСТИ:

MTTD (Mean Time to Detect):
  Формула: Время обнаружения - Время начала атаки
  Цель: < 1 час для критических
  Измерение: из временной линии инцидента

MTTR (Mean Time to Respond):
  Формула: Время первого действия - Время алерта
  Цель: < 15 минут для P1

MTTC (Mean Time to Contain):
  Формула: Время изоляции - Время начала реагирования
  Цель: < 1 час для критических

False Positive Rate:
  Формула: FP / (FP + TP) × 100%
  Цель: < 10% (слишком много FP = усталость аналитиков)

Incident Volume by Type:
  Мониторинг тенденций по типам инцидентов

Coverage (Detection Coverage):
  % техник MITRE ATT&CK, покрытых детекциями
  Цель: > 60% для основных техник
```

### TheHive — практика

```python
# Создание инцидента через TheHive API (Python)
import requests
import json

THEHIVE_URL = "http://thehive.local:9000"
API_KEY = "your-api-key"

headers = {
    "Authorization": f"Bearer {API_KEY}",
    "Content-Type": "application/json"
}

# Создание кейса
case_data = {
    "title": "INCIDENT-2024-0115: Ransomware на WS-JOHN01",
    "description": "Обнаружен ransomware. Хосты изолированы.",
    "severity": 3,          # 1=Low, 2=Med, 3=High, 4=Critical
    "startDate": 1705315020000,  # Unix timestamp ms
    "tags": ["ransomware", "p1", "malware"],
    "tlp": 2,               # 0=White, 1=Green, 2=Amber, 3=Red
    "pap": 2
}

response = requests.post(
    f"{THEHIVE_URL}/api/case",
    headers=headers,
    data=json.dumps(case_data)
)

case = response.json()
case_id = case["id"]
print(f"[+] Case created: {case_id}")

# Добавление IOC (observable)
observable_data = {
    "dataType": "ip",
    "data": "185.220.101.47",
    "message": "C2 сервер ransomware",
    "tlp": 2,
    "ioc": True,
    "tags": ["c2", "ransomware"]
}

requests.post(
    f"{THEHIVE_URL}/api/case/{case_id}/artifact",
    headers=headers,
    data=json.dumps(observable_data)
)

# Добавление задачи
task_data = {
    "title": "Форензика WS-JOHN01",
    "description": "Снять образ диска и дамп RAM",
    "status": "InProgress",
    "assignee": "maria.ivanova"
}

requests.post(
    f"{THEHIVE_URL}/api/case/{case_id}/task",
    headers=headers,
    data=json.dumps(task_data)
)

print("[+] Incident fully set up in TheHive")
```

## 📌 Итоги главы

- NIST SP 800-61 определяет 4 фазы жизненного цикла инцидента: Подготовка → Обнаружение и анализ → Сдерживание/Ликвидация/Восстановление → Действия после инцидента
- **Подготовка** — самая важная фаза: команда, инструменты, плейбуки, SLA
- **Обнаружение** требует ответов на 5 вопросов (Что/Кто/Когда/Где/Как) и построения временной линии
- **Сдерживание** — изолировать, не выключать (теряем RAM-артефакты)
- **Post-mortem** — не поиск виноватых, а поиск улучшений
- Ключевые метрики: MTTD, MTTR, MTTC, False Positive Rate
- TheHive — основная open-source платформа управления инцидентами
