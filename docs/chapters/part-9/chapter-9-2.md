## 9.2.1 Стратегия перехода: от PHP к SOC

## 9.2.6 Типичные вопросы на собеседовании SOC L1: полные ответы

### Вопрос 1: "Что такое SOC и чем занимается аналитик L1?"

**Развёрнутый ответ для собеседования:**

> SOC — Security Operations Center — это централизованное подразделение, отвечающее за непрерывный мониторинг, обнаружение и реагирование на инциденты информационной безопасности. SOC работает в режиме 24/7 и является "нервной системой" безопасности организации.
>
> Аналитик L1 — это первая линия обнаружения. Основные задачи:
> - Мониторинг алертов в SIEM-системе
> - Первичный триаж: определение, является ли алерт реальной угрозой или ложным срабатыванием
> - Базовое расследование: сбор контекста (IP-адреса, хэши файлов, временная метка)
> - Эскалация подтверждённых инцидентов на L2/L3
> - Документирование всех действий в тикет-системе
> - Соблюдение SLA по времени реагирования

**Схема уровней SOC:**
```
+----------+     Эскалация      +----------+     Эскалация      +----------+
|  SOC L1  | =================> |  SOC L2  | =================> |  SOC L3  |
|          |                    |          |                    |          |
| Триаж    |                    | Глубокий |                    | Threat   |
| алертов  |                    | анализ   |                    | Hunting  |
| Первичн. |                    | IR       |                    | Malware  |
| расслед. |                    | Forensics|                    | Analysis |
+----------+                    +----------+                    +----------+
  ~60-80% алертов                ~15-30%                          ~5-10%
  закрываются здесь
```

### Вопрос 2: "Как бы вы расследовали подозрительный алерт?"

**Структурированный ответ по шагам:**

> Я бы использовал методичный подход, основанный на принципе "не паниковать, собрать факты":

```
АЛГОРИТМ РАССЛЕДОВАНИЯ АЛЕРТА:

1. ПЕРВИЧНЫЙ ТРИАЖ (< 5 минут)
   ├── Прочитать описание алерта
   ├── Определить тип (malware, network, auth, data)
   ├── Оценить severity (Critical/High/Medium/Low)
   └── Открыть тикет в системе

2. СБОР КОНТЕКСТА (5-15 минут)
   ├── Исходный актив: что за машина? (hostname, IP, owner)
   ├── Пользователь: кто залогинен? нормальная активность?
   ├── Временная корреляция: что происходило до/после?
   └── Обогащение IoC: IP → VirusTotal, AbuseIPDB
                       Hash → VirusTotal, MalwareBazaar
                       Domain → Whois, Passive DNS

3. АНАЛИЗ И ВЕРИФИКАЦИЯ (15-30 минут)
   ├── Сравнить с baseline нормального поведения
   ├── Проверить связанные события в SIEM
   ├── Определить: True Positive или False Positive?
   └── Если TP — оценить масштаб (один хост / распространение?)

4. ДЕЙСТВИЕ
   ├── False Positive: задокументировать, закрыть тикет, создать
   │   исключение или доложить о проблеме правила
   └── True Positive: эскалировать на L2 с полным контекстом

5. ДОКУМЕНТИРОВАНИЕ
   └── Записать все шаги, находки, принятые решения
```

**Python-пример: автоматизация обогащения IoC**

```python
#!/usr/bin/env python3
"""
IOC Enrichment Script для SOC L1
Автоматически обогащает IoC через VirusTotal API
"""

import requests
import json
import sys
from datetime import datetime

VT_API_KEY = "your_virustotal_api_key"

def check_ip_vt(ip_address: str) -> dict:
    """Проверка IP-адреса через VirusTotal."""
    url = f"https://www.virustotal.com/api/v3/ip_addresses/{ip_address}"
    headers = {"x-apikey": VT_API_KEY}
    
    try:
        response = requests.get(url, headers=headers, timeout=10)
        if response.status_code == 200:
            data = response.json()
            stats = data["data"]["attributes"]["last_analysis_stats"]
            return {
                "ip": ip_address,
                "malicious": stats.get("malicious", 0),
                "suspicious": stats.get("suspicious", 0),
                "harmless": stats.get("harmless", 0),
                "country": data["data"]["attributes"].get("country", "Unknown"),
                "verdict": "MALICIOUS" if stats.get("malicious", 0) > 3 else "CLEAN"
            }
    except Exception as e:
        return {"ip": ip_address, "error": str(e)}

def check_hash_vt(file_hash: str) -> dict:
    """Проверка хэша файла через VirusTotal."""
    url = f"https://www.virustotal.com/api/v3/files/{file_hash}"
    headers = {"x-apikey": VT_API_KEY}
    
    try:
        response = requests.get(url, headers=headers, timeout=10)
        if response.status_code == 200:
            data = response.json()
            stats = data["data"]["attributes"]["last_analysis_stats"]
            name = data["data"]["attributes"].get("meaningful_name", "Unknown")
            return {
                "hash": file_hash,
                "name": name,
                "malicious": stats.get("malicious", 0),
                "total_engines": sum(stats.values()),
                "verdict": "MALICIOUS" if stats.get("malicious", 0) > 5 else "SUSPICIOUS" if stats.get("malicious", 0) > 0 else "CLEAN"
            }
        elif response.status_code == 404:
            return {"hash": file_hash, "verdict": "NOT_FOUND"}
    except Exception as e:
        return {"hash": file_hash, "error": str(e)}

def enrich_alert(alert_data: dict) -> None:
    """Обогащение алерта данными из внешних источников."""
    print(f"\n{'='*60}")
    print(f"ОБОГАЩЕНИЕ АЛЕРТА: {alert_data.get('alert_id', 'N/A')}")
    print(f"Время: {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}")
    print(f"{'='*60}")
    
    # Проверяем IP
    if "src_ip" in alert_data:
        result = check_ip_vt(alert_data["src_ip"])
        print(f"\n[IP] {result['ip']}")
        print(f"  Страна: {result.get('country', 'N/A')}")
        print(f"  Вредоносных: {result.get('malicious', 0)}")
        print(f"  Вердикт: {result.get('verdict', 'N/A')}")
    
    # Проверяем хэш файла
    if "file_hash" in alert_data:
        result = check_hash_vt(alert_data["file_hash"])
        print(f"\n[HASH] {result['hash'][:16]}...")
        print(f"  Имя файла: {result.get('name', 'N/A')}")
        print(f"  Вредоносных AV: {result.get('malicious', 0)}/{result.get('total_engines', 0)}")
        print(f"  Вердикт: {result.get('verdict', 'N/A')}")
```

### Вопрос 3: "Объясните разницу между IDS и IPS"

**Полный ответ:**

```
IDS vs IPS — ВИЗУАЛЬНОЕ СРАВНЕНИЕ:

       СЕТЬ                               СЕТЬ
         |                                  |
    +----+----+                        +----+----+
    | Firewall|                        | Firewall|
    +----+----+                        +----+----+
         |                                  |
         |          IDS                     |          IPS
    +----+----+   (КОПИЯ)              +----+----+
    | Switch  |---копия трафика--->    |  IPS    |<--- трафик проходит
    +----+----+  +----------+         +----+----+     ЧЕРЕЗ устройство
         |       |   IDS    |              |
    [Сервера]   |  Анализ  |         [Сервера]
                +----------+
                     |
                 Оповещение
                 (только!)

IDS — Intrusion DETECTION System:     IPS — Intrusion PREVENTION System:
  + Пассивный режим (зеркало трафика)   + Активный режим (inline)
  + Не влияет на пропускную способность + Может БЛОКИРОВАТЬ трафик
  + Нет риска ложных блокировок         - Ложные срабатывания = блокировка
  - Только обнаруживает, не блокирует   - Влияет на задержку сети
  
Примеры: Snort (IDS), Zeek/Bro        Примеры: Snort (IPS mode), Suricata
```

> **Дополнение:** На практике Snort и Suricata могут работать в обоих режимах. Zeek/Bro — преимущественно IDS с мощными возможностями анализа. В современных NGFW (Next-Generation Firewall) функционал IPS обычно встроен.

```
           процессы, логи ОС
```

### Вопрос 4: "Что такое false positive и как с ними работать?"

**Развёрнутый ответ:**

> **False Positive (ложное срабатывание)** — это когда SIEM или IDS генерирует алерт на нормальную, легитимную активность, ошибочно классифицируя её как угрозу.

```
МАТРИЦА КЛАССИФИКАЦИИ АЛЕРТОВ:
                    
                    Реальная угроза?
                    ДА          НЕТ
                  +----------+----------+
  Алерт      ДА  |   TRUE   |  FALSE   |
  поднят?        | POSITIVE | POSITIVE |
              НЕТ|  FALSE   |   TRUE   |
                 | NEGATIVE | NEGATIVE |
                  +----------+----------+

TRUE POSITIVE:  Алерт есть, угроза есть  → Реагируем!
FALSE POSITIVE: Алерт есть, угрозы нет  → Настраиваем правило
FALSE NEGATIVE: Алерта нет, угроза есть → Опасно! Улучшаем детект
TRUE NEGATIVE:  Алерта нет, угрозы нет  → Всё хорошо
```

**Как работать с FP:**
1. **Не игнорировать** — даже если "очевидно" FP, проверить
2. **Задокументировать** причину: "Сканирование с IP 10.1.1.5 — это Nessus, запускается каждый понедельник"
3. **Создать исключение** — в SIEM добавить whitelist/exception rule
4. **Уведомить старшего** — если FP системная (нужна корректировка правила)
5. **Отслеживать тренды** — если FP слишком много, правило неэффективно

```python
# Пример SPL-запроса для анализа качества правила в Splunk:
"""
index=security sourcetype=alerts rule_name="Suspicious Port Scan"
| stats count by src_ip, verdict
| where verdict="false_positive"
| sort -count
| head 10
"""

# Пример: если одно правило генерирует >50% FP — оно требует пересмотра
```

**Q: Объясните модель OSI и TCP/IP**

```
OSI (7 уровней)              TCP/IP (4 уровня)
====================         ====================
7. Application               4. Application
6. Presentation                 (HTTP, DNS, FTP,
5. Session                       SSH, SMTP)
--------------------         --------------------
4. Transport                 3. Transport
   (TCP, UDP)                   (TCP, UDP)
--------------------         --------------------
3. Network                   2. Internet
   (IP, ICMP, ARP)              (IP, ICMP, ARP)
--------------------         --------------------
2. Data Link                 1. Network Access
1. Physical                     (Ethernet, WiFi)
```

**Q: Что происходит при вводе google.com в браузере?**
1. DNS-запрос: браузер запрашивает IP для google.com
2. TCP три-стороннее рукопожатие (SYN → SYN-ACK → ACK)
3. TLS-рукопожатие (для HTTPS)
4. HTTP GET-запрос
5. Сервер возвращает ответ (200 OK + HTML)
6. Браузер рендерит страницу

**Q: Как отличить TCP от UDP?**

| Характеристика | TCP | UDP |
|---------------|-----|-----|
| Надёжность | Гарантированная доставка | Ненадёжный |
| Соединение | Connection-oriented | Connectionless |
| Порядок пакетов | Гарантирован | Не гарантирован |
| Скорость | Медленнее | Быстрее |
| Применение | HTTP, SSH, FTP | DNS, VoIP, видеостриминг |

### Linux-вопросы с практическими примерами

**Q: Как найти подозрительные процессы на Linux?**

```bash
# Показать все запущенные процессы с полными путями
ps auxf

# Найти процессы, слушающие порты
ss -tulpn
netstat -tulpn  # устаревший, но спросят

# Найти процессы с подозрительными соединениями
ss -tnp | grep ESTABLISHED

# Показать файлы, открытые конкретным процессом (PID 1337)
lsof -p 1337

# Найти исполняемые файлы в /tmp (подозрительно!)
find /tmp -type f -executable -ls

# Проверить cron-задачи (persistence mechanism)
crontab -l
cat /etc/cron* 2>/dev/null
ls -la /var/spool/cron/

# Проверить автозагрузку
systemctl list-units --type=service --state=enabled

# Последние входы в систему
last
lastb  # неудачные попытки входа

# Проверить sudoers (privilege escalation)
cat /etc/sudoers
sudo -l
```

**Q: Как анализировать логи SSH-брутфорса?**

```bash
# Найти неудачные попытки входа (Debian/Ubuntu)
grep "Failed password" /var/log/auth.log | tail -50

# Подсчитать попытки по IP
grep "Failed password" /var/log/auth.log | \
  awk '{print $11}' | sort | uniq -c | sort -rn | head 20

# Найти успешные входы после серии неудач (подозрительно!)
grep -E "(Failed|Accepted)" /var/log/auth.log | \
  grep -A5 "Accepted" | grep -B5 "Failed"

# Проверить текущие SSH-сессии
who
w
```

### SIEM-вопросы: Splunk SPL

**Q: Напишите SPL-запрос для обнаружения брутфорса**

```splunk
| Search для обнаружения brute force атак:

index=windows EventCode=4625
| stats count as failed_logins by src_ip, user
| where failed_logins > 10
| sort -failed_logins
| eval severity=if(failed_logins > 50, "CRITICAL", 
                   if(failed_logins > 20, "HIGH", "MEDIUM"))
| table src_ip, user, failed_logins, severity
```

**Q: Как искать аномалии в DNS-трафике (DNS tunneling)?**

```splunk
index=dns
| stats count by query_length, src_ip
| where query_length > 50
| eventstats avg(query_length) as avg_len, stdev(query_length) as std_dev
| where query_length > avg_len + (2 * std_dev)
| sort -query_length
```
